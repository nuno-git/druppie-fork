# Sandbox

Everything about the sandbox: the container image, the daemon that runs
inside it, how pi tool calls are rerouted to hit that daemon, and the
runtime choice (Kata vs Sysbox).

## Runtime choice

Two options, selected via `ONESHOT_SANDBOX_RUNTIME`:

| Runtime | What it is | Use when | Host requirement |
|---|---|---|---|
| `kata-runtime` | Kata Containers — each container is a real microVM with its own kernel | production | host must expose KVM (`/dev/kvm`); nested virt enabled on VM providers |
| `sysbox-runc` | Sysbox CE — enhanced runc with user-namespace isolation + procfs virt | dev on hosts without nested virt (e.g. Hetzner Cloud) | `sysbox-ce` installed as a Docker runtime |

Flag wiring is in
[`src/sandbox/lifecycle.ts`](../src/sandbox/lifecycle.ts) — specifically
`defaultSandboxLaunchOptions()` and `launchSandbox()`. The `--privileged`
flag is only added under Kata (scoped to the microVM, safe). Under Sysbox,
dockerd-in-container works without `--privileged`.

At startup we verify the chosen runtime is actually registered with the host
Docker daemon (`ensureRuntimeAvailable`). If it isn't, we fail loudly with
install instructions rather than silently fall back.

## The container image

Built from [`sandbox/Dockerfile`](../sandbox/Dockerfile). Base is
`debian:bookworm-slim`. Runs as root inside the container (that's safe under
Kata because VM-root ≠ host-root, and under Sysbox because the user
namespace maps container-root to an unprivileged host user).

Contents:

- **System tooling**: `git`, `curl`, `build-essential`, `python3`, `pip`,
  `ripgrep`, `fd-find`, `jq`, `openssh-client`
- **Node.js 22 LTS** (`/usr/bin/node`, `/usr/bin/npm`)
- **Python libs**: `aiohttp` (for the daemon)
- **Docker Engine** + **compose plugin** + **buildx** + **containerd.io**
  — so agents can `docker compose up` inside the sandbox
- **`/app/daemon.py`** — the exec daemon (more below)
- **`/usr/local/bin/entrypoint.sh`** — the container entrypoint

Image is ~1.5 GB (mostly Docker Engine + Node + Playwright deps).

### Entrypoint flow

[`sandbox/entrypoint.sh`](../sandbox/entrypoint.sh):

```
1. Start dockerd in the background on /var/run/docker.sock
2. Wait up to 30s for `docker info` to succeed (= dockerd ready)
3. exec python3 /app/daemon.py   (takes over as PID 1-equiv)
```

The delay in step 2 matters — if an agent tries `docker compose up` before
dockerd is up, it fails. By gating the daemon on dockerd readiness, every
agent-bash call that arrives is guaranteed to find Docker available.

## The daemon

[`sandbox/daemon.py`](../sandbox/daemon.py) — an aiohttp HTTP server
listening on `0.0.0.0:8000` inside the container. The host publishes that
port on a random loopback port (`127.0.0.1:<random>`) via Docker port
mapping.

### Authentication

Every request must carry `Authorization: Bearer <token>`. The token is a
256-bit random generated by the host at container launch
([`src/sandbox/lifecycle.ts`](../src/sandbox/lifecycle.ts) — `randomBytes(32)`),
passed via `-e SANDBOX_AUTH_TOKEN=...`, and stored only in the host's
`SandboxClient` instance. No one outside the orchestrator process can hit
the daemon.

Health check is the only unauthenticated route.

### Endpoints

Defined in [`sandbox/daemon.py`](../sandbox/daemon.py) `build_app()`:

| Method | Path | Purpose |
|---|---|---|
| GET | `/health` | readiness probe; returns `{"ok": true, "workspace": "/workspace"}` |
| POST | `/exec` | run a bash command in `/workspace`; streams stdout/stderr as SSE, ends with `exit` event |
| POST | `/read` | read a file from /workspace (utf-8 or base64 for binaries) |
| POST | `/write` | write a file (creates parent dirs) |
| POST | `/access` | check `R_OK` (or `R_OK|W_OK`) access on a path |
| POST | `/stat` | file metadata (exists, isFile, isDirectory, size, mtimeMs) |
| POST | `/ls` | directory listing |
| POST | `/find` | glob via `fd` (respects .gitignore, excludes node_modules/.git) |
| POST | `/grep` | search via `ripgrep --json` |
| POST | `/bundle` | `git bundle create /out/<name>` for the whole repo (`--all`) |
| POST | `/import-bundle` | import a uploaded bundle as /workspace state |
| POST | `/init` | configure git identity (+ optional remote URL) |

All POSTs accept JSON bodies; `/import-bundle` takes raw bytes (up to 512 MB).

### Path handling

Every path the daemon accepts (for read/write/stat/etc.) is resolved relative
to `/workspace` and verified to not escape it via `..`:

```python
def _abs_in_workspace(rel: str) -> Path:
    p = (WORKSPACE / rel).resolve()
    try:
        p.relative_to(WORKSPACE.resolve())
    except ValueError:
        raise web.HTTPBadRequest(reason=f"path escapes workspace: {rel}")
    return p
```

So even if an agent asks for `/etc/passwd`, the daemon rejects it. This is
defense-in-depth — the sandbox itself doesn't have host-file access anyway.

## Tool routing: pi + remote-ops

Pi's SDK defines tools like `read`, `bash`, `edit`, `write`. By default
these operate on the local filesystem via Node `fs` / `child_process`.

We swap out the **operations** each tool uses so they route to the daemon
instead. This is how pi's "same tool, same schema, different backend"
extension point is used.

### The ops shims

In [`src/sandbox/remote-ops.ts`](../src/sandbox/remote-ops.ts):

- `createRemoteBashOps(client)` returns a `BashOperations` whose `exec()`
  opens an SSE stream against `/exec` and forwards stdout/stderr to the
  caller.
- `createRemoteReadOps(client)` / `createRemoteEditOps(client)` /
  `createRemoteWriteOps(client)` return ops whose `readFile`/`writeFile`/
  `access` POST to `/read`, `/write`, `/access`.
- `createRemoteLsOps(client)` / `createRemoteFindOps(client)` /
  `createRemoteGrepOps(client)` are the same, except **synchronous** —
  pi's `ls`/`find`/`grep` call some methods (like `existsSync`, `statSync`,
  `readdirSync`) synchronously. The sync impl shells out to
  `curl --unix-socket ...` via `execFileSync`. Adds ~5-10 ms per call;
  fine for the low call volume.

Every path the ops see has already been resolved by pi via `resolveToCwd`
using `SANDBOX_CWD_SENTINEL = "/workspace"`. The ops strip the sentinel
prefix and pass relative paths to the daemon (which re-expands them inside
its own workspace). Absolute paths *outside* `/workspace` get rejected
locally via `toRel()` throwing. Defense in depth — the daemon would reject
them too.

### Binding to pi

Pi's SDK has **two** distinct option fields for tools, which is a historical
gotcha that cost us most of an afternoon to spot:

```ts
tools?: Tool[];                   // used ONLY to extract active-tool names
customTools?: ToolDefinition[];   // custom tool implementations; override built-ins by name
```

Passing custom tools via `tools` alone: pi reads their names, then runs its
own built-in implementations (with `process.cwd()` defaults) — our sandbox
operations get ignored entirely. This silently leaks every bash/read to the
host filesystem.

Correct usage (in [`src/sandbox/tools-factory.ts`](../src/sandbox/tools-factory.ts)):

```ts
return {
  customTools,                     // real operations
  activationTools: customTools.map(t => ({ name: t.name })),  // name stubs
};
```

Orchestrator then passes:

```ts
createAgentSession({
  tools: sandboxTools.activationTools,       // just names
  customTools: sandboxTools.customTools,     // actual ops
  ...
});
```

The `read/write/edit/ls/grep/find/bash` names in `customTools` replace the
built-in implementations by name; pi's tool registry ends up with our
sandbox ops for each.

## Lifecycle

[`src/sandbox/lifecycle.ts`](../src/sandbox/lifecycle.ts):

```
launchSandbox(opts)
  ├── ensureImage(opts.image)          // `docker image inspect`
  ├── ensureRuntimeAvailable(runtime)  // `docker info` must list the runtime
  ├── mkdtemp host-side                // for out/ bundle dir only
  ├── authToken = 32 bytes hex
  ├── docker run
  │     --runtime=<runtime>
  │     [--privileged if kata]
  │     --memory=4g --cpus=2 --pids-limit=8192
  │     -p 127.0.0.1::8000             // random host port
  │     -v <host-bundle-dir>:/out      // only shared path
  │     -e SANDBOX_AUTH_TOKEN=...
  │     oneshot-sandbox:latest
  ├── docker port container-name 8000  // find the random published port
  ├── new SandboxClient({host:'127.0.0.1', port: hostPort, authToken})
  ├── waitForHealth  (polls /health every 200ms for up to 60s)
  ├── setTimeout to SIGTERM the container after timeoutSec (24h default)
  └── return { containerName, hostPort, bundleHostDir, authToken, client, stop }
```

`stop()` cancels the wall-clock timer, runs `docker rm -f`, removes the host
tmp dir.

## What crosses the sandbox boundary

Only three things flow between host and sandbox:

1. **Auth token** — set once at container launch, used to authorize every RPC.
2. **Loopback TCP** for the daemon protocol (commands, results, streaming).
3. **The `/out` bind mount** — used exclusively for the final git bundle.
   It's the only filesystem path both sides see. Host-side, the bundle is
   treated as opaque bytes (fed to `git fetch` in the push-sandbox, never
   parsed or executed as code).

Nothing else. No workspace share (agents write to `/workspace` in the VM
only). No SSH agent forward. No environment variable forward for anything
except the auth token.
