Summary/Detail pattern: SessionSummary for lists, SessionDetail for single items. All domain exports through a single __init__.py. Repositories return domain models, never ORM objects.

03 — Orchestration

The orchestration engine

The orchestrator is a LangGraph-based loop that runs agents sequentially. A Planner agent dynamically creates and revises the execution plan after every major phase.

Orchestrator loop

  • Picks the next PENDING agent run from the plan
  • Loads the agent definition (YAML)
  • Enters the agent's tool-calling loop
  • Agent calls LLM → gets tool calls → executes → repeats
  • Agent calls done() → orchestrator moves to next run
  • If next run is Planner → Planner re-evaluates and creates new plan steps

State reconstruction

On resume (after pause, reboot, sandbox completion), the agent reconstructs its full LLM conversation from the database. Every message, tool call, and result is persisted. No in-memory state is needed.

Adaptive planning

The Planner doesn't create the full pipeline upfront. It plans a few steps, then re-evaluates:

  • Did Architect approve or send feedback to BA? → design loop
  • Did tests pass or fail? → retry loop (max 3) or HITL escalation
  • Did user approve the preview or want changes? → feature branch loop

Session status model

activeProcessing in progress
pausedStopped by user or after reboot
paused_approvalWaiting for role-based approval
paused_hitlWaiting for user answer
paused_sandboxWaiting for sandbox completion
completedAll agents finished
failedError occurred

Safety limits (per agent YAML)

max_iterations: 5 (Summarizer) to 100 (Builder). Planner: 15. The orchestrator enforces hard stops.

03 — Orchestration

Agents are YAML

Every agent is defined declaratively. No agent behavior is hardcoded. Want a new agent? Write a YAML file.

# agents/definitions/business_analyst.yaml id: business_analyst name: Business Analyst Agent description: Gathers requirements... system_prompt: | You are a senior business analyst... # Composable system prompt fragments system_prompts: - summary_relay - done_tool_format - workspace_state # Which MCP tools this agent can call mcps: coding: - read_file - write_file - list_dir # Per-agent approval overrides approval_overrides: "coding:write_file": requires_approval: true required_role: business_analyst # LLM + limits model: glm-4 temperature: 0.2 max_iterations: 50

What the YAML controls

  • System prompt (+ composable fragments)
  • MCP tool whitelist per server
  • Approval overrides (tighten global defaults)
  • LLM model, temperature, max tokens
  • Iteration safety limit
  • Skills the agent can invoke
  • Extra builtin tools (execute_coding_task)

Two-layer approval model

mcp_config.yaml sets global defaults (e.g., merge_pull_request requires developer approval). Agent YAML can tighten for its context (e.g., BA's write_file needs BA role approval, but Builder's write_file is auto-approved).

Skills: composable capability packs

Markdown files with YAML frontmatter. When invoked at runtime, the agent gets temporary access to the skill's tools + instructions. Used for: code review, git workflows, coding standards, architecture templates.

Hidden parameter injection

session_id, repo_name, user_id are injected by the platform and hidden from the LLM. Agents can't target arbitrary repos or sessions.

04 — Governance

Tool governance

Every agent action is a tool call. Every tool call can be logged, inspected, and gated. This is how we keep AI agents under control.

Tool execution flow

1. LLM returns tool call
Agent's LLM decides to call e.g. docker:build_image
2. Whitelist check
Is this tool in the agent's YAML mcps list? If not → rejected.
3. Parameter injection
Hidden params (session_id, repo_name) injected from session context.
4. Approval gate
If requires_approval: true → pause, wait for authorized role to approve/reject.
5. Execute via MCP
HTTP call to containerized MCP server. Result persisted to DB.

What gets approved (examples)

ToolRequired role
docker:build_imagedeveloper
docker:run_containerdeveloper
coding:merge_pull_requestdeveloper
coding:write_file (BA)business_analyst
coding:write_file (Architect)architect
coding:write_file (Builder)auto-approved

Full audit trail

Every tool call is persisted: who called it, which agent, arguments, result, approval status, who approved, timestamps. The Inspect view in the UI lets you drill into any call.

Mermaid validation gate

Design documents pass through a Mermaid syntax validator before the approval gate. Catches LLM mistakes (escaped quotes, wrong arrows, reserved keywords) so human reviewers don't waste time on broken diagrams.

05 — Sandbox

Isolated sandbox execution

When an agent needs to write serious code, it delegates to an isolated Docker container. The sandbox has no access to the host, no real credentials, and no network access beyond the LLM proxy and git server.

Sandbox flow

1. Agent calls execute_coding_task
Built-in tool (not MCP). Passes task description + sandbox agent name.
2. Control plane creates session
Provisions per-sandbox Gitea user, proxy keys, credential store.
3. Sandbox container starts
Fresh Docker container with OpenCode, git. Clones project from Gitea via proxy.
4. OpenCode executes autonomously
Writes code, runs tests, commits, pushes. LLM calls go through proxy.
5. Webhook callback resumes agent
HMAC-signed webhook. Fetches events, extracts file changes, resumes orchestrator.

Three sandbox agents

AgentPurpose
druppie-builderImplements features, writes code
druppie-testerWrites tests, validates, reports
druppie-core-builderDual-repo: modifies Druppie's own code

All enforce mandatory git workflow: add → commit → push. No unpushed work.

Dual-network isolation

Sandbox containers can only reach:

  • LLM proxy (rewrites model requests to real providers)
  • Git proxy (injects Gitea/GitHub credentials server-side)

No internet. No host filesystem. No other Docker containers.

Resource limits

Each sandbox: 4GB RAM, 2 CPUs (configurable). Optional Kata Containers for VM-level isolation with a separate guest kernel — each sandbox gets its own QEMU lightweight VM.

05 — Sandbox

Credential proxying

Sandboxes never see real credentials. All access goes through the control plane, which injects tokens server-side.

📂

Git proxy

Sandbox clones via proxy URL with random key. Control plane injects real Gitea password or GitHub App token. Validates owner/repo scope — requests for other repos return 403.

🤖

LLM proxy

All LLM requests go through the control plane. Injects real API keys per provider. Handles transparent failover between providers on non-2xx responses.

🔗

GitHub API proxy

Reverse proxy for api.github.com. Sandbox gets opaque $GITHUB_API_PROXY_URL with 256-bit random key. Bearer token injected server-side.

Credential lifecycle

  • Gitea: Per-sandbox user (sandbox-{id}) with write access to one repo only. Created before sandbox, deleted after.
  • GitHub: Short-lived App installation tokens (~1h TTL). Auto-expire, no cleanup needed.
  • Proxy keys: Invalidated on sandbox completion, container destroy, or session delete.
  • Orphan cleanup: On backend startup, all sandbox-* Gitea users are deleted.

The core update flow (dual-repo sandbox)

For Druppie self-improvement, a special sandbox clones two repos:

  • /workspace/core/ — Druppie's GitHub repo
  • /workspace/project/ — Project repo with FD/TD from Gitea

Creates PR on GitHub targeting colab-dev. A human developer must review and merge the PR before the pipeline continues.

05 — Sandbox

LLM provider routing

We use LiteLLM internally for standardized tool calling across providers. Each agent has its own model profile, and the proxy handles failover transparently. We're not locked into any single provider.

Transparent proxy failover

sub-second

The LLM proxy sits between every agent and the LLM providers. When a provider returns a non-2xx response (rate limit, auth error, outage), the proxy transparently retries with the next provider in the chain, rewriting the model name in the request body.

The agent never notices. From its perspective, the request just succeeded.

Agent sends request → Provider A: 429 rate limit → Proxy rewrites model to Provider B → 200 OK → Agent continues as normal

Profile-based model routing

Each agent and sandbox agent has its own model profile in sandbox_models.yaml. The LLM proxy resolves profile names to real provider chains at request time.

# sandbox_models.yaml sandbox/druppie-builder: - provider: zai model: glm-4-plus - provider: deepinfra model: Qwen/Qwen3-32B - provider: openrouter model: anthropic/claude-sonnet sandbox/druppie-tester: - provider: deepinfra model: Qwen/Qwen3-32B

Different agents can use different providers. The Builder might use Claude for code quality, while the Tester uses a cheaper model.

Force override

Set LLM_FORCE_PROVIDER + LLM_FORCE_MODEL in .env to override all profiles with a single provider. Useful for testing or when one provider is known-good.

06 — TDD Pipeline

TDD by architecture

Testing isn't optional or convention-based. It's enforced by the pipeline structure. The agents literally can't skip it.

How the TDD loop works

  • Builder Planner reads FD + TD, writes builder_plan.md with test strategy, code standards, file structure
  • Test Builder writes comprehensive test suites based on the plan. Sets up frameworks and deps. Does NOT run tests.
  • Builder reads the tests and implements code to pass them. Works in a sandbox.
  • Test Executor runs tests, diagnoses failures, fixes code in an internal retry loop. Reports structured PASS/FAIL via test_report builtin.

Failure handling

  • PASS: Planner routes to Deployer
  • FAIL (retry < 3): Planner sends back to Builder with failure feedback
  • FAIL (retry ≥ 3): Planner escalates to user via HITL with three choices:
Continue
User gives specific instructions, Builder retries
Deploy anyway
Deployer deploys with warning despite failures
Abort
Summarizer ends the workflow
07 — Module System

The module system

A Druppie module is a containerized MCP server that exposes tools via the MCP protocol. The key insight: the same module can be called by AI agents during development AND by the generated application in production.

Three categories

TypeUsed byExamples
coreAgents onlycoding, docker, filesearch
moduleApps onlyApp-specific capabilities
bothAgents + AppsOCR, classifier

Context-aware calling

Agent call: Platform injects session_id, project_id transparently. Agent never sees them.

App call: SDK passes app_id, project_id, user_id explicitly from env vars + Keycloak token.

The module doesn't care who's calling — the presence of session_id vs app_id tells the context.

Trade-off: Apps own their data and auth, but have a runtime dependency on the module platform for shared capabilities. This is a deliberate choice — it enables central governance, usage tracking, and one version to maintain.

# Module file structure druppie/mcp-servers/module-ocr/ MODULE.yaml # 3 fields: id, versions, latest server.py # Routes /v1/mcp, /v2/mcp, /mcp db.py # Module's OWN database auth.py # Keycloak JWT validation v1/ module.py # Business logic tools.py # MCP tool defs (source of truth) schema/ 001_initial.sql current.sql v2/ module.py # Independent v2 logic tools.py # Independent v2 schema schema/ 001_add_pages.sql current.sql

Define once principle

MODULE.yaml has only 3 fields (id, versions, latest). Everything else — tool names, descriptions, schemas, agent guidance — lives in code via @mcp.tool() decorators and is discoverable via MCP tools/list. Zero duplication.

07 — Module System

Versioning & storage

Simultaneous major versions

v1 and v2 run at the same time as independent codebases. No translation layer. No sunset mechanism. Path-based routing keeps them isolated.

PathRoutes to
/v1/mcpv1/tools.py
/v2/mcpv2/tools.py
/mcpLatest version
/healthAggregate health

Breaking change rules (SemVer)

New major version needed:

  • Removing/renaming a tool, parameter, or response field
  • Changing a field's type or semantics
  • Making an optional parameter required

In-place update (minor/patch):

  • Adding new tool, optional parameter, response field
  • Bug fixes, performance improvements

Module-owned storage

Modules never connect to Druppie's database. They get their own PostgreSQL container (or are stateless). Context comes through injected MCP parameters.

No schema coupling — Druppie renames a column, modules don't break
Portable modules — develop, test, run a module without Druppie's schema
Safe resets — Druppie's "reset DB" can't break modules

Module data is isolated from Druppie, but it's still hosted on the platform. Apps that call modules have a runtime dependency on the module infrastructure — like any app that uses cloud APIs.

Additive-only database changes

Both v1 and v2 share the same module DB. Strict rules:

  • Add columns (with defaults), tables, indexes — never drop or rename
  • No SELECT * — explicit columns so new fields don't break old versions
  • Every new column has a DEFAULT — older version code can INSERT without it

08 — Security

Authentication architecture

Keycloak is the single identity provider for everything: the platform, generated apps, and module MCP servers. All users exist in the druppie realm.

Who authenticates how

ComponentToken source
Platform usersKeycloak JWT via frontend login
Generated appsSame Keycloak realm, app-specific client
Sandbox agentsShort-lived OBO token (15 min TTL)
Module serversValidate JWT against Keycloak JWKS

Role-based access

RoleCan approve
adminEverything
architectArchitecture designs
developerDocker ops, PR merges
business_analystFunctional designs

Token for identity, arguments for context

These are separate concerns:

  • JWT token proves who the user is
  • Standard MCP arguments provide calling context (session_id, project_id, app_id)

Modules validate the JWT, then use injected arguments to attribute usage to the right user, project, and context.

App-level RBAC

Each generated app owns its own role system in its own database. The project template provides RBAC tables, admin page, and auth helpers out of the box.

Why in the app? Access control is app-specific. Different apps need different roles. No coupling between Druppie's DB and app-specific access data.

Note: Apps own their auth and business data, but depend on the module platform for shared capabilities (OCR, classification, etc.). This is a platform dependency

Sandbox token exchange

Before sandbox launch: Druppie requests an OBO token from Keycloak (token-exchange grant). 15-minute TTL. Carries the original user's identity (sub). Usage is attributed correctly even when an agent acts on behalf of the user.

08 — Security

Data architecture

Single PostgreSQL database with a fully relational schema. No JSON/JSONB columns — everything is normalized into proper tables. The database is the single source of truth: agents reconstruct state from DB on resume.

Session & execution

  • sessions — User requests
  • agent_runs — Per-agent execution
  • llm_calls — Every LLM interaction
  • tool_calls — Every tool invocation
  • messages — Chat timeline

Governance

  • approvals — Pending/resolved gates
  • questions — HITL interactions
  • plans / plan_steps — Agent pipelines
  • module_usage — Cost tracking

Projects & sandbox

  • projects — Created applications
  • deployments — Container deploys
  • sandbox_sessions — Ownership mapping
  • users — Keycloak-synced

Design rules

  • No migrations — Update SQLAlchemy models directly, reset DB with docker compose
  • No JSON/JSONB — If you need structured data, make a table
  • No legacy/fallback code — Clean architecture only
  • State reconstruction — On resume, full LLM conversation is rebuilt from DB records

Inspect view: full observability

Every session exposes a drill-down view:

  • Timeline of all agent runs with status
  • Expand any LLM call: prompt, response, tokens, timing
  • Expand any tool call: arguments, result, approval status, who approved
  • Sandbox sessions: files changed, events timeline, full conversation
  • Retry from any agent run (reverts git state + re-executes)

11 — Status

Where we are right now

We're being honest: this is a working platform, but it's not production-ready yet. We want it to work well before we put it in front of real users.

What works

  • Full agent pipeline: requirements → architecture → code → test → deploy
  • Governance layer: approvals, HITL, audit trail, role-based access
  • Sandbox isolation with credential proxying
  • TDD enforcement by architecture
  • Provider-agnostic LLM routing via LiteLLM
  • Session stop/resume, retry from any agent run
  • Self-improvement flow (Druppie modifying its own code via PRs)

Our approach

We're building iteratively, testing each part thoroughly before moving to the next. No shortcuts. When something doesn't work, we fix the architecture — we don't patch around it.

Real user testing comes after the core platform is solid. We'd rather ship late and right than early and broken.

What we're actively working on

  • Module system — The specification is done, implementation is in progress. This is the foundation for reusable capabilities across generated apps.
  • Coding agent quality — The sandbox agents produce working code, but reliability and code quality need to improve before we're confident showing it to end users.
  • Testing & evaluation framework — Building systematic ways to measure agent behavior: are the right agents called? Do they produce correct output? Benchmarking the pipeline end-to-end.
  • Infrastructure maturity — Moving from Docker Compose to a production-grade container orchestration (Kubernetes / managed platform).
  • UI polish — Dashboard and settings pages are prototypes. The chat and approval flows work, but the overall UX needs refinement.
