Who we are

02 — Context

What is Rijnland?

Rijnland is a water authority — a public-law government body unique to the Netherlands. We manage water levels, flood defenses, and wastewater treatment for 1.3 million people in the western Netherlands, including parts of Amsterdam, Leiden, and the surrounding polder landscape.

Our core tasks are keeping feet dry, water clean, and dikes strong. Not exactly a tech company. But running this infrastructure increasingly depends on good software — and as a public authority, everything we build must be explainable and serve the public interest.

We're not a tech company — but we need tech to do our job.

03 — Why we built this

The problem we faced

Like many public-sector organizations, we have a growing need for internal tooling — dashboards, data pipelines, operational apps — but a limited development team to build them.

We explored off-the-shelf solutions and Microsoft's low-code tools. They cover a lot, but when you need custom software — something tailored to a specific water management process — you hit a wall.

Then agentic AI entered the picture. We started asking: what if AI agents could do the structured parts of software development, while humans stay in control of the decisions that matter?

High demand
Many internal apps needed, not enough developer capacity
Custom needs
Water management processes don't fit generic low-code templates
The hypothesis
AI agents can handle structured dev work — under human oversight

What if the next developer on the team wasn't a person — but a network of agents?

04 — The Product

How software gets built

A user says "Build me a todo app". Here's what happens inside Druppie:

1. Classify Intent
Router Agent
Determines this is a "create project" request. Creates a Git repository.
2. Plan the Pipeline
Planner Agent
Decides which agents to run and in what order. Re-plans after each phase.
3. Gather Requirements
Business Analyst Agent
Asks the user structured questions (multiple choice + text). Produces a functional design doc.
4. Design Architecture
Architect Agent
Validates against standards, writes a technical design with diagrams. Can reject back to BA.
5. Plan Implementation
Builder Planner Agent
Creates a detailed code plan with standards, file structure, and test strategy.
6. Write Tests First
Test Builder Agent — TDD Red
Writes comprehensive test suites based on the plan. Does not run them yet.
7. Build the Code
Builder Agent → OpenCode Sandbox — TDD Green
Delegates to an isolated sandbox container. Writes code, commits, pushes to git.
8. Run Tests & Fix
Test Executor Agent
Runs tests, diagnoses failures, fixes code. Iterates up to 3 times, then escalates.
9. Deploy Preview
Deployer Agent
Builds Docker image, runs container, gives user a preview URL. Requires approval.
10. User Feedback Loop
Planner re-evaluates
User approves → summarize & wrap up. User wants changes → loop back.

04 — The Product

Key technical choices

Everything is YAML-configured

Every agent is defined declaratively — system prompt, tool whitelist, LLM profile, approval overrides, iteration limits. No behavior is hardcoded. Want a new agent? Write a YAML file.

MCP & built-in tools

Agents interact with the world through MCP (Model Context Protocol) tool servers and built-in tools. Every action — file write, git push, Docker build — is auditable and governable.

TDD by design

Tests are written before code. A separate agent writes tests (Red phase), another writes code to pass them (Green phase), and a third runs and fixes them. The agents can't skip testing.

Human-in-the-loop at every critical step

Every tool has an approval policy. Dangerous operations — Docker builds, merges, deployments — pause execution and wait for the right role to approve. Agents also ask structured questions to gather user input.

Skills as composable capabilities

Skills are Markdown-based capability packs agents invoke at runtime, granting temporary tool access and instructions. Code review, git workflows, diagram generation — all reusable across agents.

LLM provider resilience

Three-layer failover: proxy-level switch, Druppie-level retry with a different model chain, and a watchdog that detects stalled sessions. We're not locked into a single provider.

05 — Challenges

The elephant in the room

We manage water. We run pumps and dikes and treatment plants. We have never in our history built and shipped software as a product. And now we're doing exactly that.

That forces a set of questions we’ve never had to answer before — questions that are second nature to software companies but completely new to us. And as a public authority, we can’t just move fast and break things — we need to be able to justify and explain what we ship.

We went from ‘we use software’ to ‘we build software’ — and those are very different things.

New territory

How do you write good software?

What does quality look like when you don't have decades of engineering culture? How do you enforce standards when you're still figuring out what the standards should be?

Security

How do you keep it safe?

Supply chain attacks, dependency management, vulnerability scanning, secrets handling — we're a government body running AI agents that write and deploy code. The security bar is high and we're learning as we go.

Infrastructure

How do you set up the infra?

CI/CD pipelines, container orchestration, monitoring, logging, environments — all things we had to figure out from scratch. We're running 15+ Docker services. That's a lot for a water authority.

05 — Challenges

Technical & organizational challenges

Beyond the identity question, these are the specific walls we're hitting. We're sharing them openly — because we think the interesting conversations start here.

Technical

Context window limits

Complex projects generate massive context across 12 agents. Keeping each agent informed without blowing token limits is a constant balancing act.

Technical

Agent reliability

LLMs can be unpredictable. An agent might misinterpret a plan, skip a step, or loop endlessly. We've built guardrails, but reliability is an ongoing effort.

Technical

Testing AI behavior

How do you write tests for a system whose behavior is nondeterministic? Our TDD pipeline helps for generated code, but validating the orchestration layer is genuinely hard.

Organizational

Public sector & AI governance

As a public authority, building an AI platform raises questions that private companies rarely face — data sovereignty, procurement rules, legal compliance, public accountability, and the need for transparency and explainability in how AI-generated software serves the public interest.

Feasibility

Where's the ceiling?

Druppie handles well-structured apps today. Complex, stateful, or UI-heavy applications stretch current capabilities. Are we building a useful tool or chasing a moving target?

Sustainability

Maintaining what we build

Building software is one thing — maintaining it is another. Patching, upgrading, monitoring, on-call. Can a small team inside a water authority sustain a platform this complex long-term?
