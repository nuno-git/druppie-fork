# module-coding

**Port:** 9001. **Type:** core. **Dockerfile:** `druppie/mcp-servers/module-coding/Dockerfile`.

The workhorse — every file operation, git operation, test run, and PR creation flows through here. 21 tools.

## Dockerfile

- Base: `python:3.11-slim`.
- Installs: `git, curl, nodejs, npm, chromium`.
- Puppeteer config: `PUPPETEER_EXECUTABLE_PATH=/usr/bin/chromium`, `PUPPETEER_SKIP_CHROMIUM_DOWNLOAD=true`.
- `npm install -g @mermaid-js/mermaid-cli` — used by `make_design` to validate Mermaid before writing.
- Workspace dir: `/workspaces` (mounted from `druppie_new_workspace` volume).
- Expose 9001.

## `v1/module.py` security

Blocks dangerous command patterns at `run_git` / shell invocation time:
- `rm -rf /`, `mkfs`, `dd`, `sudo`, `chmod 777`
- Shell reverse-shell payloads (`nc -e`, `bash -i >& /dev/tcp/`)
- Backtick subshells that resolve to root paths

Sanitises LLM-supplied string params: the literal strings `"null"`, `"none"`, `"undefined"` become Python `None` before execution.

Workspace registry:
- In-memory dict keyed by `session_id` → workspace path.
- Persisted to `.druppie_state.json` inside each workspace.
- On restart, workspaces are rediscovered from disk.

## Git configuration

- `GITEA_URL` = `http://gitea:3000`
- `GITEA_ORG` = `druppie`
- `GITEA_TOKEN`, `GITEA_USER`, `GITEA_PASSWORD` from env.
- All git operations use HTTPS clone URLs with the token.

## Long-running tools

`run_tests`, `install_test_dependencies` have `1200 s` timeouts configured in `druppie/execution/tool_executor.py:LONG_RUNNING_TOOLS`. Every other tool uses the default `60 s`.

## Workspace path

Each session gets `/workspaces/<session_id>/`. Inside: git clone of the Gitea project repo, a hidden `.druppie_state.json`, and any design files the agents write.

When a session ends (COMPLETED/FAILED), the workspace is left on disk. Volume retention is until the next `reset-hard` or manual cleanup.

## Mermaid validation

`make_design` and `validate_design` extract fenced code blocks with `mermaid` language and pipe each through `mmdc` (mermaid-cli). Failures:
- Syntax error → tool returns `success: false, error: "<details>"`
- mmdc crash → tool still returns error, doesn't crash the MCP server

This is why business_analyst and architect can rely on make_design to catch Mermaid typos before the design is committed.

## Integration with Gitea

- Clone URL pattern: `https://<user>:<token>@gitea:3000/<org>/<repo>.git`
- PR creation: Gitea REST API `POST /api/v1/repos/{owner}/{repo}/pulls`
- Merge: `POST /api/v1/repos/{owner}/{repo}/pulls/{number}/merge`
- Status of PRs / branches: Gitea REST API

The Gitea token is the admin PAT generated by `setup_gitea.py`. For true multi-user setup, each user's `user_tokens` row would be used instead; that path is wired but not yet exercised.
