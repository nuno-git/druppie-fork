# Local Control Plane

Path: `background-agents/packages/local-control-plane/`. Stack: TypeScript, Express.js, `better-sqlite3`, WebSocket.

Port: **8787** on both `druppie-new-network` and `druppie-sandbox-network`. The only service dual-homed on both networks.

## Responsibilities

1. Accept session-create requests from Druppie (via the sandbox integration).
2. Proxy LLM requests from sandboxes to configured providers (with failover + retries).
3. Stream events from sandboxes back to subscribers (Druppie, web UI, bots).
4. Persist sessions and events to SQLite for replay.
5. Call Druppie's webhook on completion with an HMAC signature.

## HTTP surface

Key routes in `router.ts`:

- `POST /sessions` вЂ” create a session, spawn a sandbox, return `{session_id, ws_url, llm_proxy_url}`.
- `GET /sessions/{id}/events` вЂ” paginated event log.
- `POST /sessions/{id}/stop` вЂ” cancel the sandbox.
- `DELETE /sessions/{id}` вЂ” mark archived.
- `ALL /llm-proxy/{proxyKey}/{provider}/*` вЂ” LLM proxy (detailed in `llm-proxy.md`).

## WebSocket

`GET /ws/session/{id}` upgrades to WebSocket. Server-pushed events:
- `{type: "tool_call", ...}`
- `{type: "tool_result", ...}`
- `{type: "token", content: "вЂ¦"}` (streaming LLM tokens)
- `{type: "error", ...}`
- `{type: "git_sync", ...}`
- `{type: "completed", ...}`

Druppie does NOT subscribe to this WebSocket directly. It relies on:
- Periodic `/api/sandbox-sessions/{id}/events` polling via the Druppie backend (which caches snapshots).
- The `/complete` webhook at session end.

## SQLite adapter

Tables:
- `sessions` вЂ” id, repo, user, status, started_at, completed_at.
- `events` вЂ” session_id, sequence, type, payload (JSON), created_at.
- `credentials` вЂ” proxyKey, session_id, provider, apiKey (in-memory only вЂ” never persisted).

Durable data lives under `DATA_DIR=/data` (the `druppie_sandbox_data` volume). Credentials are in-memory so a control-plane restart invalidates them (sandboxes must re-negotiate).

## Startup

```
node_modules/.bin/tsx packages/local-control-plane/src/index.ts
```

Env vars:
- `PORT` вЂ” default 8787.
- `DATA_DIR` вЂ” SQLite path, default `/data`.
- `INTERNAL_API_SECRET` вЂ” HMAC secret for sandbox-manager handshakes.
- Various LLM provider URLs baked in at build time.

## Webhook to Druppie

At session completion:
1. Construct the result payload.
2. Compute HMAC-SHA256 of the body using `webhook_secret` (generated by Druppie during registration).
3. POST to `{DRUPPIE_URL}/api/sandbox-sessions/{session_id}/complete` with `X-Druppie-Signature: <hmac>`.
4. Retry 3Г— with exponential backoff on non-2xx.

If all retries fail, the sandbox data is still persisted locally and Druppie's watchdog will eventually mark the tool call FAILED.

## Network dual-homing

The control plane is on both networks because:
- Druppie backend (on `druppie-new-network`) calls it to create sessions.
- Sandboxes (on `druppie-sandbox-network`) call it for LLM proxying and event emission.
- Sandboxes CANNOT reach Druppie directly вЂ” they only talk to the control plane.

This is the security boundary вЂ” a compromised sandbox can only hit the control plane's limited API.
