# update_core_builder

File: `druppie/agents/definitions/update_core_builder.yaml` (119 lines).

## Role

Implements changes to Druppie's own codebase ‚Äî new MCP modules, agent definition tweaks, backend fixes. Unlike the regular `builder`, this agent targets the Druppie repo (`/workspace/core/`), not a user project.

Outputs a PR against the Druppie fork's `colab-dev` branch. Does NOT merge (developer must review).

## Dual-repo sandbox layout

When `execute_coding_task(agent="druppie-core-builder", repo_target="druppie_core")` is called, the sandbox is set up with:
- `/workspace/core/` ‚Äî the Druppie GitHub fork, read+write.
- `/workspace/project/` ‚Äî the user's project repo, read-only context.

This lets the sandbox understand what the user is building while modifying Druppie itself.

## Mandatory parameters

Every `execute_coding_task` call from this agent MUST include:
- `agent="druppie-core-builder"` (sandbox persona).
- `repo_target="druppie_core"` (not `"project"`, which would write to the user's repo).

## Task content

The task prompt instructs the sandbox to:
1. Create a branch `core/<description>` from `colab-dev` (not from `main` ‚Äî `main` is deprecated per `CLAUDE.md`).
2. Implement the change (new module, agent update, bug fix).
3. If adding a new MCP module, follow `druppie/skills/module-convention/SKILL.md`:
   - Create `druppie/mcp-servers/module-<name>/` with full structure.
   - Add `Dockerfile`, `requirements.txt`, `MODULE.yaml`, `server.py`, `v1/module.py`, `v1/tools.py`.
   - Add service to `docker-compose.yml`.
   - Add entry to `druppie/core/mcp_config.yaml`.
4. Commit with conventional messages.
5. Push branch.
6. Create PR targeting `colab-dev` (NOT `main`).
7. Return summary with PR URL ‚Äî do NOT merge.

## Approval gate

Global `coding:merge_pull_request` requires `developer` approval. Additionally this agent's `done()` requires `developer` approval ‚Äî the developer reviewing the PR is the one who closes out the agent run. This prevents the agent from "finishing" on its own when a human hasn't yet confirmed the PR is acceptable.

## GitHub App auth

Cloning + PR creation against the Druppie GitHub fork uses a GitHub App installation token generated by `druppie/services/github_app_service.py`. Env vars:
- `GITHUB_APP_ID`
- `GITHUB_APP_PRIVATE_KEY_PATH` ‚Äî PEM file mounted into the backend.
- `GITHUB_APP_INSTALLATION_ID`

The token is minted per sandbox, passed through to the sandbox via the control plane, and used to clone + push + open PR.

## `DRUPPIE_REPO_OWNER` / `DRUPPIE_REPO_NAME`

Env vars identify which GitHub repo to target:
- `DRUPPIE_REPO_OWNER=nuno-git` (fork owner).
- `DRUPPIE_REPO_NAME=druppie-fork` (repo name).

Changing these points the agent at a different fork ‚Äî useful for evaluation environments.

## Why not auto-merge

Modifying Druppie itself is high-risk: it affects every future session. The design is "agent proposes, human disposes" ‚Äî the developer reviews the PR, possibly requests changes, merges when satisfied.

## After merge

There's no automatic redeploy. The developer triggers the Druppie stack restart (probably via CI in production). The new module or updated agent becomes available on next backend startup (ToolRegistry re-fetches schemas).
