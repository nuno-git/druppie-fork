01 — Overview

The Planner: the brain

The Planner runs after every agent. It reads the previous agent's output, detects signals, and decides what happens next.

Rule: one agent at a time

Every plan has exactly 2 steps: one working agent + Planner re-evaluation. Exception: Summarizer is always alone (terminal).

Loop prevention

Max 30 Planner iterations. After that, forces finalization: merge → deploy → summarize.

Signal-based routing

Agents signal the Planner via keywords in their done() summary:

  • DESIGN_APPROVED → proceed to next phase
  • DESIGN_FEEDBACK → loop back to BA
  • DESIGN_REJECTED → abort
  • DESIGN_APPROVED_CORE_UPDATE → core flow
  • TEST RESULT: PASS → deploy
  • TEST RESULT: FAIL → retry or escalate
  • USER FEEDBACK: ... → improve or finalize

Summary relay

When an agent calls done(), all previous agent summaries are collected and prepended to the next agent's prompt. This is how agents coordinate without direct communication.

Retry strategies (escalating)

  • Retry 1: Targeted fixes — minimal surgical changes per failing test
  • Retry 2: Rethink & rewrite — re-read TD, rewrite failing components
  • Retry 3: Simplify — strip to simplest code that passes. Hardcode if needed.

HITL escalation (after 3 failures)

BA asks the user (in Dutch):

  • Continue with guidance → user gives instructions, Builder retries
  • Deploy anyway → Deployer deploys with test failure warning
  • Abort → Summarizer ends the project

Builder works in sandbox

Every Builder run delegates to execute_coding_task. Code is written by OpenCode (open-source AI coding agent) inside an isolated Docker container managed by background-agents. Commits and pushes via git proxy. Builder pulls the changes.

When does this trigger?

Architect detects the project modifies Druppie's own codebase: agent definitions, prompts, skills, MCP servers, workflows.

Dual-repo sandbox

The sandbox clones two repos:

  • /workspace/core/ — Druppie's GitHub repo (read+write)
  • /workspace/project/ — project repo with FD/TD (read-only)

Creates branch core/<name> from colab-dev, commits, pushes, creates PR.

Key: Architect runs twice

1st: "This project needs core changes" → triggers core update
2nd: "Core changes merged, now design the actual project" → normal flow

A human developer must review and merge the GitHub PR before the pipeline continues.

05 — Gates

All gates at a glance

Approval gates

Agent pauses. Authorized role must approve or reject.

  • BA writes functional_design.md
    business_analyst role approves
  • Architect writes technical_design.md
    architect role approves
  • Docker compose up
    developer role approves
  • Merge pull request
    developer role approves
  • Update Core Builder calls done()
    developer role approves (PR review)

HITL questions

Agent pauses. User answers via chat UI.

  • Router: "Is this correct?" (confirm intent)
  • BA: Clarifying questions (multiple choice, 1 at a time)
  • BA: "Does this FD capture your needs?" (confirm design)
  • Deployer: "App is live at URL. Does this look good?"
  • BA (escalation): "Tests failed 3x. Continue / deploy anyway / abort?"
  • BA/Architect (general_chat): Answer user questions

Sandbox pauses

Agent pauses while sandbox container executes. Resumes automatically via webhook.

  • Builder: execute_coding_task (implementation)
  • Developer: execute_coding_task (improvements)
  • Update Core Builder: execute_coding_task (core changes)
05 — Gates

Signal reference

Business Analyst signals

  • DESIGN_APPROVED → Architect
  • NO_FD_CHANGE → Architect (validate only)
  • DESIGN_REJECTED → Summarizer (abort)
  • CHAT_COMPLETE → Summarizer
  • CHAT_ROUTE_AGENT → different agent
  • CHAT_ROUTE_CREATE_PROJECT → create flow
  • CHAT_ROUTE_UPDATE_PROJECT → update flow

Architect signals

  • DESIGN_APPROVED → Builder Planner
  • DESIGN_APPROVED_CORE_UPDATE → Core Builder
  • DESIGN_FEEDBACK → BA (revision loop)
  • DESIGN_REJECTED → Summarizer (abort)
  • CHAT_COMPLETE → Summarizer
  • CHAT_ROUTE_AGENT → different agent

Test Executor signals

  • TEST RESULT: PASS → Deployer
  • TEST RESULT: FAIL → Builder (retry < 3) or BA (escalation)

Deployer signals

  • USER FEEDBACK: (positive) → Summarizer (or Developer merge)
  • USER FEEDBACK: (changes) → Developer (improvements)

How it works

Agents include these signal strings in their done() call. The Planner detects keywords in the summary relay and routes accordingly. No hardcoded routing — it's all prompt-driven.
